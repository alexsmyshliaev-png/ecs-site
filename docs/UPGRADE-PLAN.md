## Ключевое архитектурное решение (ADR)

**2026-07-07 · Отказ от собственного бэкенда в пользу CRM-форм Битрикс24.**

Контекст: FastAPI-бэкенд существовал ради двух задач — (1) скрыть вебхук Битрикс24 от браузера при отправке лидов, (2) собственная БД событий аналитики. Аудит показал: для прода бэкенду нужны VPS, деплой, rate limiting, ретраи, мониторинг, бэкапы БД с ПДн — несоразмерный ops-объём для корпоративного сайта.

Решение: заявки принимаются **CRM-формами Битрикс24** (виджет). Они закрывают всё то же без сервера:

- лид создаётся в CRM напрямую, никаких секретов в браузере (вебхук не используется вообще);
- антиспам встроен: Yandex SmartCaptcha (по умолчанию) или reCAPTCHA v2;
- UTM-метки подхватываются формой автоматически;
- согласие на обработку ПДн (152-ФЗ) — встроенное поле формы;
- скрытые поля + JS-API (`b24:form:init`, `form.setProperty()`, `form.setValues()`) — для передачи product/page/utm_first/ym_client_id;
- аналитика — Яндекс.Метрика (цели уже размечены через ecsTrack), собственная events-БД не нужна.

Следствия: сайт — чистая статика (хостинг проще и дешевле), каталог `backend/` архивируется, из `leads.js` удаляются все сетевые пути.

Критерии возврата к бэкенду (пересмотреть решение, если появится): свой личный кабинет клиента, серверные расчёты, интеграции сверх CRM, потребность в собственном хранилище событий для BI.

Документация (Claude Code: сверяйся с ней, не с памятью):
- Хаб инструментов: https://github.com/bitrix-tools/bitrix24-dev-hub
- REST-документация: https://apidocs.bitrix24.ru/ (есть MCP-сервер: https://apidocs.bitrix24.ru/ai-tools/mcp.html — подключи в Claude Code для точных ответов по API)
- Размещение CRM-формы на стороннем сайте: https://helpdesk.bitrix24.ru/open/13003062/

## Этап 1 — Миграция заявок на CRM-формы Битрикс24

### 1.1 Подготовка в портале Битрикс24 (делает владелец, не Claude)

- [ ] Создать CRM-формы (CRM → Интеграции → CRM-формы), минимум две:
  - **«Заявка» (универсальная)** — поля: Имя*, Телефон*, Email, Комментарий; скрытые поля: `product`, `form_slug`, `page_url`, `utm_first`, `ym_client_id`, `referrer_first`.
  - **«Запись в офис»** — те же + желаемая дата/время.
  - (Опционально по одной форме на продукт — если менеджерам удобнее разные очереди.)
- [ ] В каждой форме включить: защиту от спама (Yandex SmartCaptcha), поле согласия с ссылкой на `privacy.html`, автозаполнение UTM.
- [ ] Настроить дизайн формы под бренд: фон белый, акцент #E60023, шрифт Inter, скруглённые кнопки.
- [ ] Скопировать embed-коды форм (тип «в контенте» и «по клику») и ID форм → передать в `config.js` (см. 1.2).
- [ ] Виджет на сайт (чат + обратный звонок): CRM → Интеграции → Виджет на сайт → URL загрузчика → `bitrixWidgetUrl` в config.js.
- [ ] Отключить/удалить входящий вебхук, созданный для старого бэкенда (он больше не нужен — минус один секрет).

### 1.2 Frontend: замена самописного пайплайна лидов

Файлы: `frontend/assets/js/config.js`, `leads.js`, `bitrix.js`, `ui.js`, все `*.html`.

- [ ] В `config.js`: удалить `apiUrl` и `bitrixWebhookUrl`; добавить секцию
  `b24forms: { lead: {id: "", code: ""}, office: {id: "", code: ""} }` (id/code из embed-кода форм).
- [ ] Переписать `leads.js` → модуль интеграции с CRM-формами:
  - убрать fetch на `/api/leads`, прямой вебхук и localStorage-фолбэк (ПДн в localStorage запрещены);
  - слушать `b24:form:init` и через `form.setProperty()` заполнять скрытые поля:
    `product` (из `data-product` страницы/кнопки), `form_slug`, `page_url` (location.href без PII),
    `utm_first` (из `ECS.utm.get().first`, JSON-строкой), `referrer_first`,
    `ym_client_id` (через `ym(ID, 'getClientID', cb)` — асинхронно, до сабмита);
  - события формы (init/просмотр/отправка — актуальный список сверить в helpdesk-статье 12853296) транслировать в `ecsTrack`: `lead_form_open`, `lead_success`;
  - маску телефона к полям виджета не применять (у Б24 своя валидация) — код маски оставить только если останутся собственные не-лидовые поля, иначе удалить.
- [ ] Модалка заявки: кнопки `[data-modal="lead"]` теперь открывают CRM-форму «по клику» (click-тип embed). `#leadModal` с самописной формой удалить из всех страниц (правка в `index.html` + прогон `build_pages.py`).
- [ ] Формы в теле страниц (`form.js-lead-form` на kasko/property/business/office) заменить на inline-embed CRM-формы в тех же секциях. Вёрстку секций (заголовки, буллеты, `.lead`-обёртку) сохранить.
- [ ] Фолбэк недоступности CDN Битрикса: если контейнер формы пуст через 5 с — показать блок «Не загрузилось — позвоните {phone} или напишите в WhatsApp/Telegram» + `ecsTrack("b24form_failed")`.
- [ ] Проверить, что `bitrix.js` (загрузчик виджета) не дублирует загрузчик форм.

Приёмка: тестовая заявка с каждой страницы создаёт лид в CRM с заполненными UTM, product, page_url, ym_client_id; спам-бот отсекается капчей; при отключённом интернете до b24 CDN виден фолбэк с телефоном; в консоли нет ошибок.

### 1.3 Вывод бэкенда из эксплуатации

- [ ] Переместить `backend/` в ветку `archive/fastapi-backend` (или каталог `archive/`), из `main` удалить. История в git сохраняется; в ADR — ссылка.

- [ ] `docs/api/` удалить или пометить архивными; `docs/architecture.md` — обновить потоки лида/аналитики под новую схему.
- [ ] `analytics.js`: удалить `sendToBackend()` и упоминания `/api/events` — события идут только в dataLayer + Метрику.

Приёмка: `git grep -l "apiUrl\|/api/leads\|/api/events\|bitrixWebhookUrl" frontend/ docs/` — пусто (кроме CHANGELOG/аудита).

## Этап 2 — Публикация: хостинг, безопасность, юридика

- [ ] **Боевой домен + хостинг с HTTP-заголовками** (GitHub Pages не умеет заголовки; калькуляторы Финуслуг привязаны к домену партнёра). Рекомендация: статик-хостинг с поддержкой заголовков (Cloudflare Pages / VPS-nginx / рег.ру и аналоги для РФ). Настроить:
  - `Content-Security-Policy` (allowlist: self, mc.yandex.ru, *.bitrix24.ru CDN-домены форм/виджета, finuslugi.ru, agents.finuslugi.ru, polis812.ru, googletagmanager.com при GTM);
  - `Strict-Transport-Security`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy`;
  - `X-Frame-Options: SAMEORIGIN` (или frame-ancestors в CSP).
- [ ] github.io-версию закрыть от индексации (meta noindex в деплое Pages или отключить Pages после переезда).
- [ ] **Cookie-баннер согласия** до инициализации Метрики (Вебвизор = сбор ПДн): лёгкий самописный баннер; до согласия `initMetrika()` не вызывать. Обновить `privacy.html`: Метрика/Вебвизор, CRM-формы Битрикс24 как обработчик, cookie.
- [ ] Заменить заглушки: телефон, email, адрес, фото (office, travel), координаты офиса + ключ Яндекс.Карт, `calc.travel` id (список — docs/setup.md «Что заменить перед запуском»).
- [ ] `404.html` в бренд-стиле.

Приёмка: securityheaders.com — оценка A; заявки и калькуляторы работают на боевом домене; Метрика не грузится до согласия.

- [ ] **Self-host шрифта Inter**: скачать woff2 (400/600/700/800), положить в `assets/fonts/`, `@font-face` с `font-display: swap` в tokens.css, `preload` двух основных начертаний, удалить ссылки на fonts.googleapis.com со всех страниц. - [ ] Калькуляторы — модуль `assets/js/calc.js`: разметка страниц сводится к `<div data-calc="osago|mortgage|travel">`; все id — в `config.calc` (сейчас ипотечный захардкожен в property.html); загрузка скрипта провайдера лениво по IntersectionObserver; таймаут → фолбэк «рассчитаем вручную» + `ecsTrack("calc_widget_failed")`; `preconnect` к доменам провайдеров на страницах с калькуляторами.
- [ ] Зарегистрировать сайт в Яндекс.Вебмастере и Google Search Console, отправить sitemap.

Приёмка: PageSpeed mobile ≥ 90; валидатор Schema.org без ошибок; все страницы имеют canonical/OG; шрифт грузится со своего домена.

## Этап 4 — Аналитика и маркетинг

- [ ] GTM-контейнер: сниппет на все страницы; `ecsTrack` уже пишет в dataLayer — не менять. Правило: Метрика инициализируется в коде (analytics.js, после согласия), все прочие теги (пиксели VK/MyTarget, ретаргетинг, коллтрекинг) — только через GTM.
- [ ] `utm.js`: починить `propagate()` — собирать URL через `new URL(a.href)` + `searchParams.set()` (сейчас метки дублируются и ломают якоря); дополнительно сохранять `yclid`/`gclid` в first/last touch (нужны для офлайн-конверсий Директа).
- [ ] Достроить события: `form_start` (первый фокус в CRM-форме, если API позволяет — иначе по клику на контейнер), scroll depth 25/50/75/90 (GTM-триггер), `calc_open`/`calc_complete` (postMessage от Финуслуг/polis812 — уточнить формат у партнёров).
- [ ] Цели в Метрике (reachGoal = имена событий): `lead_success` (макро), `calc_complete` (макро), `click_phone`, `fab_*`, `lead_form_open`, `calc_open` (микро). В Директе привязать конверсии к `lead_success`.
- [ ] Телефон/email/адрес выводить из config/шаблона, не хардкодом (сейчас телефон в HTML 27+ раз) — без этого невозможен коллтрекинг с подменой номера.

Приёмка: в Метрике видны все цели с тестовыми срабатываниями; в лиде CRM сходятся utm_first/utm_last/ym_client_id; GTM Preview показывает события dataLayer.

## Что сознательно НЕ делаем (и почему)

- **Свой бэкенд / БД лидов** — см. ADR выше. Возврат — по критериям из ADR.
- **React/SPA** — контентный сайт со сторонними виджетами; ADR от 2026-06-11 в силе.
- **Свой антиспам/капча** — встроен в CRM-формы.
- **Коллтрекинг, A/B, сквозная аналитика** — после стабильного рекламного трафика, через GTM (код готов).
- **Postgres/CMS** — триггеры не наступили (см. аудит §11).
