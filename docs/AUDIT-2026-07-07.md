# Технический аудит сайта ЕЦС — 2026-07-07

Объект: `Общее/ЕЦС общий сайт` (frontend — 9 статических HTML + ванильный JS, backend — FastAPI/SQLite, деплой фронтенда — GitHub Pages).

---

## Резюме

Проект в **хорошем состоянии для своей стадии**. Что сделано правильно (и это редкость для проектов такого размера):

- Бэкенд с чистой слоистой архитектурой: routers → services → models/schemas, зависимости через DI, секреты в `.env`, есть тесты (7 шт., зелёные).
- Паттерн «лид сначала в свою БД, потом в CRM» — заявки не теряются при падении Битрикс24, статус пересылки хранится (`bitrix_status`).
- Единая точка аналитики `ecsTrack()` → dataLayer + Метрика + собственный `/api/events`. Это правильная абстракция: новые системы аналитики подключаются без изменения бизнес-логики.
- UTM first/last touch реализован и передаётся в Битрикс24.
- Документация в `docs/` (архитектура с ADR, setup, API, changelog) — большинство коммерческих проектов этого не имеют.
- Осознанное решение «без React» зафиксировано в ADR — для контентного сайта со сторонними виджетами это верный выбор.
- SEO-база на месте: уникальные title/description, один `<h1>` на страницу, `lang="ru"`, alt у всех изображений, семантические `<main>/<section>/<article>/<nav>/<footer>`.

**Но публиковать в текущем виде как коммерческий сайт нельзя.** Три категории блокеров:

1. **Потеря заявок.** С пустым `apiUrl` в `config.js` (как сейчас) все заявки с продакшена уходят в `localStorage` браузера посетителя — бизнес их никогда не увидит, а посетитель видит «Спасибо!». Плюс `leads.js` не проверяет `response.ok` — ошибка 422/500 тоже выглядит как успех.
2. **Защита API отсутствует.** `POST /api/leads` и `/api/events` без rate limiting и антиспама: один скрипт зальёт БД и CRM мусором (лимит Битрикс24 ~2 rps усугубляет).
3. **Хостинг GitHub Pages** не позволяет выставить HTTP security headers (CSP, X-Frame-Options, HSTS), не даёт custom domain «из коробки» в текущей конфигурации, и калькуляторы Финуслуг привязаны к партнёрскому домену. Для рекламы и SEO нужен свой домен и хостинг с контролем заголовков.

Также: Python 3.9 — EOL с октября 2025, зависимости не зафиксированы, нет CLAUDE.md, нет robots.txt/sitemap/canonical/OG/Schema.org/favicon/404, нет cookie-баннера при включённом Вебвизоре (152-ФЗ).

Полный roadmap — в разделе 15.

## 1. Архитектура

### Текущая структура

```
frontend/          9 HTML-страниц (обвязка дублируется, синхронизация build_pages.py)
  assets/css/      tokens.css → base.css → components.css   ✅ хорошее разделение
  assets/js/       config → utm → analytics → bitrix → leads → ui → main  ✅ модули по ответственности
  build_pages.py   разноска общих блоков из index.html
  build_share.py   автономная demo-сборка
backend/app/       routers → services → models/schemas/core/dependencies  ✅ каноничный FastAPI
docs/              architecture (ADR), setup, api/, frontend/, changelog  ✅
.github/workflows/ деплой фронтенда на Pages
```

### Оценка

**Бэкенд — образцовый для своего размера.** Разделение ответственности соблюдено: HTTP-слой не содержит логики, сервисы не знают про HTTP, интеграция с Битрикс24 изолирована в `bitrix_service.py`. Clean Architecture в разумной, не догматичной степени.

**Фронтенд — главный архитектурный риск: самодельная шаблонизация.**

`build_pages.py` вырезает шапку/меню/футер из `index.html` строковыми операциями (`text.find("<!-- ===================== FOOTER")`) и вставляет в 8 других файлов. Проблемы:

- **Неявный инвариант**: общие блоки надо править только в `index.html`, потом запускать скрипт. Разработчик (или Claude в новой сессии) правит футер в `kasko.html` напрямую — следующий запуск `build_pages.py` молча затирает правку. Судя по докстрингу, это уже случалось («так пропадали кнопки»).
- Парсинг HTML регулярками и маркерами-комментариями ломается от невинного рефакторинга (переименовали комментарий FOOTER — скрипт падает или, хуже, вставляет не туда).
- `index.html` — одновременно продуктовая страница и мастер-шаблон. Смешение ролей.
- Скрипт не запускается в CI: рассинхрон страниц никак не детектируется.

**Рекомендация (главная архитектурная):** перейти на статический генератор с настоящими шаблонами, сохранив текущий HTML/CSS/JS почти без изменений. Сравнение вариантов:

| Вариант | Плюсы | Минусы | Вердикт |
|---|---|---|---|
| Оставить build_pages.py | ничего не менять | хрупкость, дрейф, не масштабируется до 20+ страниц | только до первых 2–3 новых страниц |
| **Eleventy (11ty)** | Nunjucks-шаблоны, выход — те же статические HTML, миграция = вынести header/footer в include, ~1 день | появляется Node в тулчейне | **лучший выбор**: минимальное изменение, максимальный эффект |
| Astro | компоненты, встроенная оптимизация картинок, sitemap-плагин | больше концепций, соблазн тащить JS-фреймворк | хорош, если планируется 20x рост и блог |
| Jinja2 + свой скрипт | без Node, Python уже есть | опять самодельный тулчейн без экосистемы (sitemap, минификация) | компромисс, если Node неприемлем |

С Eleventy: `_includes/header.njk`, `_includes/footer.njk`, `_data/site.json` (телефон, адрес — см. ниже про дублирование), страницы остаются HTML с фронтматтером. `build_pages.py` удаляется. Сборка в CI перед деплоем.

### Прочее

- **Два источника правды для контактов**: телефон `8 800 123-45-67` захардкожен в HTML 27+ раз (шапка/футер каждой страницы) и одновременно лежит в `config.js`. Смена номера = правка 9 файлов. Это же заблокирует коллтрекинг (подмена номера). Вынести в шаблонные данные/конфиг.
- **Нет 404-страницы** (`404.html` для GitHub Pages / hosting).
- **Кириллица и завершающий пробел в пути проекта** (`ЕЦС /Общее/ЕЦС общий сайт`) — регулярно ломает тулинг (shell-квотинг, Docker-контексты, CI checkout). Переименовать в `ecs-site` до роста команды.
- `.gitignore` исключает весь `.claude/` — командные настройки и разрешения не версионируются. Правильно: игнорировать только `.claude/settings.local.json`.

## 3. Качество кода

Общая оценка: **читаемость высокая**, файлы короткие, докстринги объясняют «почему», нейминг консистентный. KISS/YAGNI соблюдены. Конкретные проблемы:

**🔴 leads.js — успех при ошибке (потеря заявок).**

```js
return fetch(CFG.apiUrl + "/api/leads", {...})
  .catch(function (err) { ... return fallback(lead); });
// .then(): box.classList.add("is-done")  ← "Спасибо!" показывается ВСЕГДА
```

`fetch` не отклоняется на HTTP 4xx/5xx — ответ 422 (невалидные данные) или 500 показывает посетителю «Спасибо, свяжемся». Нужно: `if (!res.ok) throw ...`, состояние ошибки в UI («не отправилось — позвоните»), и повторная попытка.

**🔴 localStorage-фолбэк вводит в заблуждение.** Заявка в `localStorage` посетителя недоступна бизнесу — это не «ни одна заявка не потерялась», а «заявка потеряна, но все думают, что нет». Как временный буфер с ретраем при следующем визите — годится; как терминальный фолбэк с показом «Спасибо!» — нет.

**🟠 utm.js `propagate()` — три бага:**
- `a.href += "?" + qs` при каждой загрузке — метки дублируются в ссылках при навигации (`?utm_source=x&utm_source=x`);
- ссылка с якорем `page.html#faq` превращается в `page.html#faq?utm_source=…` — якорь ломает query;
- условие исключения внешних ссылок `a.getAttribute("href").indexOf("http") === 0` проверяется после `a.host !== location.host` с `return` — относительные внешние протоколы не покрыты, а `a[href="/"]` на Pages-подпути ведёт не туда.

Fix: собирать URL через `new URL(a.href)`, использовать `url.searchParams.set()`.

**🟠 esc() в ui.js не экранирует кавычки** — `String(t).replace(/&/g,"&amp;").replace(/</g,"&lt;")` вставляется в атрибуты (`href="..."`). Данные из config (контролируемые), но однажды туда попадёт внешний источник. Экранировать `"` и `'`.

**🟡 ui.js — 253 строки, 10 обязанностей** (шапка, reveal, FAQ, модалки, fab, контакты, toggles, tabs, аккордеоны, insurers). Пока терпимо, но fab (60 строк SVG-строк) и insurers просятся в отдельные модули. Правило на будущее: один файл — одна зона UI.

**🟡 Дублирование логики Битрикс-полей** — маппинг `fields`/`fm` собирается и в `backend/bitrix_service.py`, и в `frontend/leads.js` (фолбэк). После удаления фронтового вебхук-фолбэка (см. §8) дублирование исчезнет само.

**🟡 auth.py — сравнение токена через `!=`** — использовать `secrets.compare_digest` (timing-safe).

## 4. Компонентная архитектура

React-компонентов нет — «компоненты» здесь = HTML-блоки + CSS-классы + JS-инициализаторы. Модель здравая:

- ✅ Поведение навешивается через data-атрибуты (`data-track`, `data-modal`, `data-toggle`, `data-tabs`) — декларативно, переиспользуемо, понятно Claude.
- ✅ CSS-слои tokens/base/components — чистая иерархия.
- ❌ Крупные HTML-блоки (hero, карточки продуктов, CTA-секции) дублируются между страницами копипастой — при переходе на Eleventy станут include/макросами с параметрами.
- ❌ FAQ-данные в инлайн-`<script>` каждой страницы (`window.ECS_FAQ`) — рабоче, но контент смешан с разметкой; в SSG — фронтматтер/JSON + серверный рендер (заодно решит SEO FAQ, §6).
- ❌ Блоки калькуляторов вставлены сырыми `<script src>` в трёх вариантах на четырёх страницах — нужен единый `calc.js`-модуль (см. §10).

## 5. Производительность

База отличная: JS ~30 КБ суммарно без фреймворков, CSS ~30 КБ, изображения немногочисленны. Что чинить:

1. **Google Fonts render-blocking + недоступность Google CDN части РФ-пользователей.** `fonts.googleapis.com` в критическом пути на каждой странице; у пользователей с блокировками Google — таймаут и FOUT/задержка LCP. **Self-host Inter** (woff2, `font-display: swap`, preload двух основных начертаний). Это самый большой выигрыш по LCP/FCP для РФ-аудитории.
2. **Сторонние скрипты калькуляторов загружаются сразу и синхронно** (`iframe.bundle.iife.js` на index/osago без defer, polis812 на travel). Это чужие сотни КБ в критическом пути. Лечение — фасад: грузить по IntersectionObserver при доскролле до блока или по клику «Рассчитать» (см. §10).
3. **PNG-логотипы страховых** (84 КБ Ренессанс, 48 КБ Альфа...) — конвертировать в WebP/AVIF (~5–10x меньше) или взять SVG; добавить `loading="lazy"` (сейчас 0 на сайте) и явные `width/height` для защиты CLS.
4. **Нет минификации/фингерпринтинга ассетов** — при переходе на SSG добавить минификацию HTML/CSS/JS и хэши в имена → `Cache-Control: immutable`.
5. Мелочи: `preconnect` к `finuslugi.ru`/`polis812.ru` на страницах с калькуляторами; `scroll`-хендлер шапки дёргает `classList.toggle` на каждый скролл — обернуть в rAF (косметика).
6. CWV сейчас измерить нельзя (нет прода с трафиком) — после запуска подключить мониторинг реальных метрик через Метрику + PageSpeed CI-проверку.

SSR/SSG: контент уже статический — SSG (Eleventy) закрывает вопрос. SSR не нужен.

- Проверить, что github.io-версия **закрыта от индексации** (noindex), пока не запущен боевой домен — иначе он станет каноничным дублем.

## 7. Аналитика, маркетинг, отслеживание

Архитектурно уже правильно: `ecsTrack()` — единая шина, dataLayer-first. Новые системы = новый подписчик, бизнес-логика не трогается. Это надо сохранить и оформить правилом в CLAUDE.md.

### Что достроить

**GTM как контейнер (сейчас отсутствует).** Рекомендуемая схема: в коде остаётся только GTM-сниппет + `dataLayer.push` из `ecsTrack()`; Метрика, GA4, пиксели ВК/MyTarget, ретаргетинг-теги, коллтрекинг — тегами внутри GTM. Прямые `initMetrika()/initGA4()` из analytics.js убрать (либо оставить Метрику в коде, а всё остальное — в GTM; выбор зафиксировать). Это выполняет требование «подключать новые системы без изменения кода».

**Метрика**: init уже с `clickmap, webvisor, trackLinks, accurateTrackBounce` ✅ — карты кликов/скроллинга/Вебвизор включатся при заполнении `yandexMetrikaId`. Но Вебвизор = запись сессий → требуется согласие на обработку (cookie-баннер, §8/152-ФЗ).

**События — что уже есть**: page_view, lead_submit, lead_modal_open, faq_open, click_phone, click_email, fab_*, tab_switch, acc_open, insurer_open, toggle_open + любой `data-track`. Хорошее покрытие. **Чего не хватает:**

- `form_start` (первый фокус в поле) и `form_abandon` (ушёл, не отправив) — незавершённые заявки;
- `lead_success` / `lead_error` раздельно (сейчас `lead_submit` шлётся до ответа сервера — конверсия завышается на величину ошибок);
- calc_open / calc_interact / calc_complete — слушать `postMessage` от iframe Финуслуг/polis812 (уточнить у партнёров формат событий; у Финуслуг есть партнёрские события);
- scroll depth (25/50/75/90) и время на странице — стандартные GTM-триггеры;
- клики по мессенджерам уже трекаются через `data-track="fab_*"`, но прямые ссылки в hero (`a[data-soc]`) — проверить разметку `data-track` на всех.

**Client ID в лид.** Сейчас в лид уходят UTM+referrer+page, но не `ym_client_id` / GA client_id. Без них не построить сквозную аналитику и не отправить офлайн-конверсии («лид стал сделкой») обратно в Метрику/Директ. Добавить в `leads.js`: `ym(ID, 'getClientID', cb)` → поле лида → в Битрикс UF-поле. Это ключевой недостающий элемент для рекламы.

**UTM**: first/last touch ✅, повторные визиты ✅ (last перезаписывается, first нет), передача в CRM ✅ (last, с фолбэком на first). Доработки: attribution window (first-touch без TTL вечен — обычно 90 дней), фиксировать также `gclid`/`yclid` (авторазметка Директа и Ads — без них коллтрекинг и офлайн-конверсии не сматчатся), чинить `propagate()` (§3).

## 8. Безопасность

| # | Проблема | Риск | Детали / решение |
|---|---|---|---|
| S1 | Нет rate limiting и антиспама на `POST /api/leads`, `/api/events` | 🔴 High | CORS не защищает (curl его игнорирует). | S5 | Сторонние скрипты калькуляторов имеют полный доступ к DOM/localStorage | 🟠 Med | В `localStorage` лежит `ecs_leads` — **имена и телефоны клиентов**, читаемые любым сторонним скриптом (finuslugi, polis812, виджет Б24, будущие пиксели). ## 9. Интеграция с Bitrix CRM

Сильная часть проекта: ✅ вебхук на сервере; ✅ save-first (заявка в SQLite до отправки); ✅ статусы pending/sent/failed/disabled; ✅ современный `crm.item.add` (entityTypeId=1) вместо устаревшего `crm.lead.add`; ✅ UTM/мультиполя fm корректно; ✅ таймаут 10 с; ✅ логирование через `logger.exception`.

Доработки:

1. **🟠 Failed-лиды остаются failed навсегда.** Нет ретрая: упал Битрикс на 5 минут — лиды за эти 5 минут вручную никто не перешлёт. Решение: фоновый ретрай с экспоненциальной задержкой. Достаточно простого: FastAPI `BackgroundTasks` + периодическая задача (APScheduler) «взять все failed/pending старше N минут, попробовать снова, максимум 5 попыток» + поле `bitrix_attempts`.
2. **🟠 Нет алертинга.** `bitrix_status=failed` виден только в БД. Добавить уведомление (Telegram-бот владельцу / email) при failed и при недоступности Битрикса.
3. **🟡 Нет дедупликации** — двойной клик или повторная отправка = два лида в CRM. Идемпотентность: client-side UUID заявки + уникальный индекс, и/или в Битриксе включить контроль дублей по телефону.
4. **🟡 В CRM не передаются** `referrer`, `utm_first`, `page` отдельными полями (страница и форма склеены в comments). Завести UF-поля в Битриксе: utm_first_*, referrer, ym_client_id (§7) — без этого сквозная аналитика в CRM не сойдётся.
5. **🟡 docs/api/leads.md** упоминает `crm.lead.add` — рассинхрон с кодом, поправить.
6. Лимиты Битрикса (~2 rps) учтены в docs ✅; после добавления ретрая учесть их в背off-логике.

## 10. JS-калькуляторы страховых

Сейчас: Финуслуги ОСАГО (index, osago — синхронный `<script src>` в body), Финуслуги ипотека (property — динамическая вставка), polis812 ВЗР (travel — `<script>` с килобайтом data-атрибутов). Ипотечный `data-id` захардкожен в property.html, хотя в config.js есть секция `calc` (частичный рассинхрон конфигурации).

Проблемы: изоляции нет (S5 — доступ к localStorage с PII), производительность (§5 — синхронная загрузка), фолбэк «не загрузился» удалён с index/travel (событие `calc_widget_failed` больше не ловится), разметка размазана по 4 страницам в 3 разных стилях.

**Целевая реализация — модуль `calc.js`:**

```
<div data-calc="osago"></div>   ← вся разметка страницы
```

- `calc.js` читает `ECS_CONFIG.calc[type]` (все id — в конфиге, включая ипотеку), знает провайдера и способ вставки;
- грузит виджет лениво: IntersectionObserver (доскролл) или клик по кнопке-фасаду «Рассчитать онлайн»;
- таймаут загрузки → фолбэк-форма «рассчитаем вручную» + событие `calc_widget_failed` (вернуть!);
- события calc_open/calc_complete через postMessage → `ecsTrack` (§7);
- смена провайдера = правка одного модуля и конфига, страницы не трогаются. Это закрывает «возможность замены провайдера» полностью.

Изоляция: если провайдер позволяет iframe-встройку — предпочесть её; для script-встроек — вынести PII из localStorage (заявки-фолбэк держать в памяти/sessionStorage без имени-телефона, лучше — просто не хранить, см. S2), и понимать, что CSP должен явно допускать их домены.

## 11. Масштабируемость (сайт ×20 через 2 года)

Узкие места в порядке срабатывания:

1. **build_pages.py** — умрёт первым (уже описано, §1). С SSG 200 страниц собираются так же, как 9.
2. **Контент в HTML** — SEO-масштабирование (страницы под запросы «ОСАГО для такси», «КАСКО Санкт-Петербург»...) потребует контентных коллекций: markdown + шаблон в Eleventy/Astro. Headless CMS — только если контент начнут править не-разработчики.
3. **SQLite** — переживёт десятки тысяч лидов, но: единственный writer, файл на диске одного сервера, бэкап = копия файла. Триггер миграции на Postgres: второй инстанс бэкенда или >~10 rps событий. Подготовка сейчас: alembic + отсутствие SQLite-специфики (уже ок).
4. **Таблица events растёт бесконечно** — нет индексов (`event`, `created_at`) и политики retention. Добавить индексы сейчас, retention (агрегировать и чистить сырые события старше N месяцев) — при росте.
5. **Отсутствие CI-качества** — при 2+ разработчиках без линтеров/тестов в CI дрейф стиля и регрессии гарантированы. GitHub Actions: ruff + mypy + pytest + сборка фронта на каждый PR.
6. **Один продукт = одна страница-простыня** — при 20x нужны шаблоны «продуктовая страница» с данными, а не 20 рукописных HTML.

Что НЕ надо менять сейчас: FastAPI-структура выдержит 20x без переделки; ecsTrack-шина масштабируется; дизайн-токены готовы к росту UI.
